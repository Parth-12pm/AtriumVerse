## Models to Create (Teach Each Before Writing)

### 1. app/models/device.py
Concept: Every browser = a "device." Public key stored here lets others encrypt for this
device. Private key never appears here. is_trusted=False means the device is registered but
cannot receive any keys until an existing trusted device approves it via the ceremony.
webauthn_credential_id stores the WebAuthn credential so the server can verify future
assertions from this device during the approval flow.

Fields: id, user_id (FK→users CASCADE), public_key (TEXT), webauthn_credential_id (TEXT nullable),
device_label (VARCHAR nullable), is_trusted (BOOLEAN default False), created_at

Add to User model: devices = relationship("Device", back_populates="user")

### 2. app/models/device_link_request.py
Concept: Holds pending device link requests for both the real-time path (old device tab open)
and the async path (old device offline). Two-key design: temp_public_key is a throwaway
keypair generated by the new device for this ceremony only — it is discarded after transfer.
encrypted_private_key is null until the old device approves. expires_at is non-negotiable.

Fields: id, requesting_user_id (FK→users CASCADE), new_device_id (FK→devices CASCADE),
temp_public_key (TEXT), new_device_label (VARCHAR nullable),
encrypted_private_key (TEXT nullable), status (VARCHAR default "pending"),
approved_by_device_id (FK→devices nullable), created_at, expires_at (TIMESTAMP NOT NULL)

### 3. app/models/channel_encryption.py
Concept: Tracks per-channel encryption state and current epoch. The is_enabled flag means
unencrypted channels are completely untouched. Epoch increments when channel membership
changes — removed members get no key for the new epoch.

Fields: channel_id (PK, FK→channels CASCADE), is_enabled (BOOLEAN default False),
current_epoch (INTEGER default 1), epoch_rotated_at (TIMESTAMP)

### 4. app/models/channel_device_key.py
Concept: One row per device per channel per epoch. The encrypted_channel_key blob is the
raw channel_key wrapped via ECDH+AES-GCM between the distributing device and this device.
epoch is part of the unique constraint — without it you could not store both epoch 1 and
epoch 2 keys for the same device on the same channel.
ROWS ARE NEVER DELETED — they are the permanent record of who had access during which epoch.
Deleting them makes historical messages permanently undecryptable for that device.

Fields: id, channel_id (FK→channels CASCADE), device_id (FK→devices CASCADE),
epoch (INTEGER), encrypted_channel_key (TEXT), created_at
UniqueConstraint: (channel_id, device_id, epoch)

### 5. app/models/dm_device_key.py
Concept: One DirectMessage maps to multiple ciphertext rows — one per device that should
be able to read it (all of recipient's trusted devices + all of sender's trusted devices).

CRITICAL — FK behavior on device_id must be SET NULL, not CASCADE.
Teach this carefully: if CASCADE is used and a device is deleted, its ciphertext rows
silently disappear. The frontend then cannot distinguish between two completely different
situations:
  a) device_id IS NULL + deleted_device_id IS NOT NULL
     → "This device was removed — message was encrypted for it but the device is gone"
  b) device_id IS NULL + deleted_device_id IS NULL
     → "Message predates this device — no row was ever created for it"
These must show different UI messages. CASCADE collapses both into identical nulls and
permanently loses that information.

When a device is deleted, a separate cleanup job should:
  SET device_id = NULL, deleted_device_id = <the deleted device's id>

Fields: id, dm_id (FK→direct_messages CASCADE),
device_id (FK→devices SET NULL, nullable),
deleted_device_id (UUID nullable),
encrypted_ciphertext (TEXT nullable), created_at
UniqueConstraint: (dm_id, device_id) WHERE device_id IS NOT NULL

### 6. app/models/dm_epoch.py
Concept: Tracks the current epoch per DM conversation. DM epoch increments when either
party in the conversation links a new device. New messages use the current epoch. Old
messages carry their epoch value on the row. Without this, when key history changes,
the frontend has no way to distinguish "wrong key" from "corrupted message."

Store conversations canonically: always insert with the smaller UUID as user_a_id to avoid
duplicate rows for the same pair in opposite order. Teach the developer to enforce this at
the application layer before insert.

Fields: id, user_a_id (FK→users), user_b_id (FK→users),
current_epoch (INTEGER default 1), updated_at (TIMESTAMP)
UniqueConstraint: (user_a_id, user_b_id)

### 7. Additive Columns on Existing Models

app/models/message.py — add:
  ciphertext (TEXT nullable)
  epoch (INTEGER nullable)       ← which epoch key encrypted this, stored on message row
  is_encrypted (BOOLEAN default False)

app/models/direct_message.py — add:
  ciphertext (TEXT nullable)
  epoch (INTEGER nullable)       ← per-conversation epoch at time of send
  is_encrypted (BOOLEAN default False)

Teach: existing content column stays NOT NULL. Encrypted messages write "[encrypted]" to
content as a placeholder. The epoch column on direct_messages is essential — it tells the
recipient's browser which key derivation iteration to use when decrypting. Without it there
is no way to know which epoch's shared secret to attempt on an old message.
